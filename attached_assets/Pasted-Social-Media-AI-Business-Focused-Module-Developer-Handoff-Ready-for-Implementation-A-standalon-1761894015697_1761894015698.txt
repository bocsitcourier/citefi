1. Goals & Non-functional Requirements

Primary objectives

Allow business users to input topic/title/location/tone/mood/industry and generate multiple platform-ready posts.

Include SEO & GEO optimizations (local keywords, geo-hashtags, BLUF, short-form & platform-tailored copy).

Generate hero/platform images concurrently and attach them to posts.

Preview, edit, schedule, auto-share to connected social accounts.

Track costs (tokens), errors, and provide admin recovery tools.

Non-functional

Latency: aim to generate a batch (3 variants × 5 platforms) in under ~90–180s for typical load with concurrency controls.

Scalability: horizontal scaling on DigitalOcean; stateless API workers; independent worker processes.

Reliability: retries with exponential backoff; automatic fallback images; persistent logs.

Security: encrypt social tokens, secrets in env vars, role-based admin.

SEO/GEO: follow BLUF, H2/H3 style guidance, friendly professional tone, geo-entities included.

5. Async generation & image pipeline

We recommend NOT using Redis/BullMQ if user previously requested avoidance; however the system must still queue tasks reliably. Options:

Option A — Internal lightweight queue (recommended if small scale)
Use in-memory queue (persist to DB status) + worker pool across multiple instances with leader election (e.g., DB row lock). Good for up to hundreds tasks; ensure idempotence and persistent job state.

Option B — Serverless bursts
For heavy bursts, spawn isolated worker processes (Docker tasks) via App Platform or serverless functions; they pull jobs from DB by status = 'queued' and FOR UPDATE SKIP LOCKED to avoid races.

Below is a DB-backed job loop using PostgreSQL locking (no Redis):

Job runner pseudo:

API inserts job_batch + posts rows with status = 'PENDING'.

Worker process (cron or perpetually running) does:

BEGIN;
SELECT id FROM posts WHERE status='PENDING' LIMIT 1 FOR UPDATE SKIP LOCKED;
-- If row found, mark as RUNNING, COMMIT
-- Process: generate text (call LLM), generate images (call image API), update DB fields
-- On success: set status=COMPLETE; on failure: status=FAILED, store error


This SKIP LOCKED approach gives safe distributed workers without external queue.

Concurrency & throttling

Throttle LLM calls per worker using p-limit (e.g., 3–5 LLM calls concurrently for images).

Use per-provider rate windows. Maintain a token bucket per worker or process to avoid 429s.

Image pipeline (concurrent per post)

After draft text returned, create one or multiple image prompts.

Call image API with limited concurrency (p-limit(4)).

Stream response if supported and write to temporary file using stream API.

Upload to DO Spaces (use aws-sdk v3 S3 client with Spaces endpoint).

Persist hero_image_url and variant image_url.

Retry/backoff

Wrap all external calls in retry logic (exponential backoff + jitter). Example: retry(tries=3, backoff=2s * attempt).

After N retries, mark FAILED and emit a notification for admin or queue regeneration.

Streaming & UI progress

Use WebSocket or SSE: worker emits events generation:started, generation:section_complete, image:done, post:ready.

UI listens and updates progress bars and thumbnails live.

6. AI prompt strategy & templates (business-focused)

Principles

BLUF first: short clear first-sentence answer.

Use Q/A H2/H3 style for extended posts when generating long captions or LinkedIn articles.

Include location and local entities; prefer natural long-tail queries.

Tone/mood presets map to temperature and phrasing.

Prompt variables

$TITLE, $TOPIC, $SUMMARY, $LOCATION, $TONE, $MOOD, $INDUSTRY, $SERP_TARGET, $COMPETITOR_HEADINGS

Example prompt: generate variants (multi-platform)
SYSTEM:
You are a senior content strategist writing short, business-focused social posts. Output JSON.

USER:
Given:
- title: "$TITLE"
- topic: "$TOPIC"
- summary: "$SUMMARY"
- location: "$LOCATION"
- tone: "$TONE" (Professional|Authoritative|Friendly-Professional|Executive|Insightful)
- mood: "$MOOD" (Motivational|Analytical|Informative|Persuasive|Neutral)
- industry: "$INDUSTRY"
- intent: "$SERP_TARGET" (FeaturedSnippet|PAA|List)

Produce 3 variants for each platform in JSON. For each variant include:
- platform
- caption (max 280 chars for X; recommended length for others)
- hashtags: an array of 8–20 hashtags combining SEO + GEO
- emojis: 0–3 emojis appropriate to business tone
- suggested image prompt (short)
- suggested alt_text
- immediate CTA (1 sentence)
- email_hyperlink: for every hashtag, provide a mailto link to user email encoded as `mailto:USER_EMAIL?subject=Hashtag%20Inquiry&body=I'm%20interested%20in%20${encodeURIComponent(TITLE)}`
Return only JSON.

Example output snippet (one variant)
{
 "platform":"linkedin",
 "variant":1,
 "caption":"BLUF: Same-day courier services reduce SLA breaches by 62% for Boston retailers. Data-driven routing + local hubs shrink lead times. Read more: https://example.com/article",
 "hashtags":["#BostonLogistics","#SameDayDelivery","#RetailOps","#LastMile"],
 "emojis":["📦","📈"],
 "image_prompt":"Professional photo of a courier van in Boston Back Bay delivering packages at sunrise; business signage visible; high contrast.",
 "alt_text":"Courier van delivering in Boston Back Bay at sunrise.",
 "cta":"Read full report",
 "email_hyperlink":"mailto:user@example.com?subject=Boston%20Logistics%20Inquiry&body=I'm%20interested%20in%20..."
}


Note: System must validate hashtag safety and length.

8. Storage, CDN, and serving images

Local storage (dev)

Save images to /public/hero/<slug>.png for Next.js static serving.

Production storage

DigitalOcean Spaces (S3-compatible): upload with public-read, use CDN fronting via Spaces CDN or custom CDN.

Ensure CORS allows your frontend domain.

Serving

For external URLs (Spaces), configure next.config.js images.domains to include your space domain.

For local files, serve via /hero/<slug>.png.

Recommended pattern

Save locally (temp).

Upload to Spaces.

Save the Spaces URL in DB (hero_image_url).

Delete temp local file if storage is constrained.

Image optimization

Process images via sharp:

Convert to WebP for faster delivery when supported.

Generate platform-specific sizes.

Use Content-Type, cache headers, and versioning (?v=ts) for cache busting.

9. Auth, security, and permissions

Authentication

App auth: JWT (HttpOnly secure cookies) recommended for SSR flows.

OAuth for social platforms: store access & refresh tokens encrypted (AES-256) in DB.

Authorization

users.role supports user, editor, admin.

Admin endpoints check role === 'admin'.

Secrets

Store API keys and DO Space keys in DigitalOcean App Platform environment variables.

Never return sensitive tokens to frontend.

Input validation

Validate all incoming user input server-side. Escape HTML in captions on render unless explicitly sanitized.

Rate controlling

Apply per-user request limits to avoid abuse (e.g., 5 batch jobs / day by default).

Implement per-IP rate limits on generation endpoints.

Audit

Log admin actions in admin_action_logs (user id, action, timestamp).

10. Monitoring, logging & cost tracking

Monitoring

Use structured logs (Pino/Winston) with log levels and request context (user_id, batch_id).

Export logs to centralized system (LogDNA, Papertrail, Grafana Loki).

Metrics

Count: number of generations, images, shares.

Latency: average LLM response time, image generation time.

Errors: external API failures, quota 429s, upload failures.

Cost tracking

For each AI call, log model, tokens_input, tokens_output, estimated_cost into token_usage table.

Compute Cost_Per_Rank = token_cost / rank_improvement and store for admin report (requires rank tracking integration).

Health checks

/api/health returns DB and Spaces connection status.

12. Deployment (DigitalOcean App Platform + Docker)

Container

Dockerfile builds Node/Next.js app. Use multi-stage build:

App Platform

B. Sample DB DDL (Postgres trimmed)
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash TEXT,
  role VARCHAR(50) NOT NULL DEFAULT 'user',
  created_at TIMESTAMP DEFAULT now()
);

CREATE TABLE batches (
  id SERIAL PRIMARY KEY,
  user_id INT REFERENCES users(id),
  core_topic VARCHAR(255),
  status VARCHAR(50),
  num_requested INT,
  competitor_urls JSONB,
  created_at TIMESTAMP DEFAULT now()
);

CREATE TABLE posts (
  id SERIAL PRIMARY KEY,
  batch_id INT REFERENCES batches(id),
  topic VARCHAR(255),
  title VARCHAR(255),
  slug VARCHAR(255) UNIQUE,
  location VARCHAR(255),
  tone VARCHAR(50),
  mood VARCHAR(50),
  industry VARCHAR(100),
  status VARCHAR(50),
  seo_keywords JSONB,
  geo_tags JSONB,
  hero_image_url TEXT,
  final_text TEXT,
  created_at TIMESTAMP DEFAULT now()
);

CREATE TABLE post_variants (
  id SERIAL PRIMARY KEY,
  post_id INT REFERENCES posts(id),
  platform VARCHAR(50),
  variant_index INT,
  caption TEXT,
  hashtags JSONB,
  image_url TEXT,
  status VARCHAR(50),
  scheduled_at TIMESTAMP
);

CREATE TABLE token_usage (
  id SERIAL PRIMARY KEY,
  user_id INT,
  batch_id INT,
  model_name VARCHAR(50),
  tokens_used INT,
  estimated_cost NUMERIC(10,4),
  created_at TIMESTAMP DEFAULT now()
);
