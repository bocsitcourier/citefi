BATCH B — P0 — Auth Security

T105: P0 — Login rate-limit + account lockout

Blocked By: []
VERIFY FIRST: Confirm the login route (/api/auth/login or NextAuth credentials handler) has NO rate limiting. You rate-limited reset endpoints but (per the prior spec) not login — this is a credential-stuffing hole.

Files:


app/api/auth/login/route.ts (or the credentials authorize fn) (MODIFY)
lib/db/schema.ts — add loginAttempts tracking OR reuse loginHistory (migration if needed)


Details:


Per-IP rate limit: e.g., 10 attempts / 15 min → 429.
Per-account lockout: after N (e.g., 5) consecutive failures for an email, lock for a cooldown (e.g., 15 min) or require reset; reset the counter on success. Track via loginHistory/loginAttempts (email, ip, success, createdAt).
Do not leak which factor failed (don't reveal "email exists"). Generic "invalid credentials."
Log lockouts to activityLogs (action='account_locked'). Optionally email the user on lockout ("unusual sign-in attempts").
Apply the same per-IP limiting to the accept/invite and integrations-connect endpoints.

BATCH D — P2 — Polish & Correctness

T109: P2 — Fix refund idempotency key

Blocked By: [your existing T006]
Bug: `idempotencyKey: refund-chargeId−{chargeId}-
chargeId−{Date.now()}` defeats idempotency — a retry generates a new key and can double-refund.
Fix: Use a deterministic key tied to the intended operation, e.g., `refund-chargeId−{chargeId}-
chargeId−{amount}` or a server-generated refundRequestId created once and reused on retry. File: app/api/admin/billing/refund/route.ts (MODIFY).

DEFER — Do NOT build before paying customers

These are real eventually, but building them now is gold-plating an admin panel for users you don't have yet. Explicitly out of scope for launch:


Admin impersonation / "log in as user" — useful for support later, real security surface, not now.
Global audit-log viewer (cross-user adminActionLogs browser) — your prior spec's per-user activity is enough for launch.
Bulk admin actions (bulk suspend/role/delete) — premature with a handful of users.
SSO / SAML / SCIM — defer until you actually sell the Agency "Custom"/enterprise tier; that's what those buyers mean by "enterprise support."



Done Looks Like (launch bar)


An agency can invite teammates, they accept and get the right role (T101).
An agency can connect a publishing account and see/revoke it; secrets are encrypted, never returned to the client (T102).
An agency can create and switch between client workspaces, and content/usage/integrations are isolated per workspace (T103).
An end client can log in as an approver and review/approve content read-only (T104).
Login is rate-limited and locks out brute-force attempts; the reset flow uses one coherent mechanism (T105, T106).
The billing UX shows usage, a settable hard cap, alerts, and a pre-job estimate — no surprise bills (T107).
A user can export their data and delete their account, and deletion cancels Stripe + revokes credentials + cascades (T108).
Refund idempotency is deterministic; cancel/refund/invite send confirmation emails (T109, T110).
New agencies get a ≤3-step onboarding (T111).
The DEFER list is not built.

Pre-launch, must-have: BATCH A (T101–T104) + BATCH B (T105–T106) + T107 + T108. This is the real launch bar — an agency literally cannot use the product without A, and B/C protect you legally and competitively.

The point of all of this is still to get ONE real local-SEO agency using it. Build A+B, get a design partner in, and let their feedback reprioritize C and D — don't build the full list in a vacuum.
