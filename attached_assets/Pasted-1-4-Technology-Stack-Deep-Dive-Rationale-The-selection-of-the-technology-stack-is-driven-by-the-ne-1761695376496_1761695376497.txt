AI API Integration and ResilienceThe Worker Service must treat all external API communication as volatile, requiring comprehensive error handling and retry logic.3.2.1. Robust Retry Mechanism: Implementing Exponential BackoffExternal API calls are susceptible to transient network failures, quota limitations, and server-side rate limiting (HTTP 429). An immediate retry often fails; therefore, an exponential backoff strategy is essential to prevent system overload and maximize job success rates.23The implementation involves wrapping every external AI API call within a loop that checks for specific failure conditions (like 429 or 5xx errors). If a recoverable failure occurs, the system must await a delay period before retrying. This delay should increase exponentially with each attempt—for example, $2^n$ seconds, where $n$ is the retry attempt number.23For professional resilience, developers should avoid manual Promise wrapping and instead mandate the use of established Node.js libraries designed for exponential backoff, such as p-retry or node-backoff.23 These libraries efficiently manage jitter (randomized delay to prevent synchronization issues) and enforce maximum wait times, ensuring that API calls are retried safely without blocking the Node.js event loop.3.2.2. AI Token Usage Tracking and LoggingAccurate cost logging is paramount for the financial accountability of the platform.OpenAI (GPT-4) Token Extraction: When using the non-streaming Chat Completion endpoint (stream: false), the OpenAI API reliably returns usage statistics within the response payload.21 The Worker must access the usage property within the final API response object to extract the prompt_tokens and completion_tokens.
